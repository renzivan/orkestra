Orkestra — Implementation Plan

Goal: A local web app where a user defines reusable skills, projects, models, and agents, composes agents into linear flows, then runs a task through a flow (or single agent) by spawning local CLIs and chaining their output.

Architecture: Next.js (App Router) on Bun, single process. SQLite (bun:sqlite) at ~/.orkestra/orkestra.db holds all definitions and run history. A template engine turns each agent's model into an argv, spawns the CLI (input via stdin, result via stdout), and a flow runner chains steps, retries on failure, and streams live output to the browser over SSE.

For agentic workers: this plan is implemented task-by-task with TDD. Each task ends in an independently testable, committable deliverable. Steps use - [ ] tracking.

✓ Resolved decisions (from grilling)

  • Purpose: multi-agent coordination; runs agents by spawning local CLIs.
  • Topology: linear pipeline (Flow = ordered agents; output → next input).
  • Agent = base instruction (required) + 0..N skills + 0..N projects + 1 model.
  • Skill = name + markdown body only. Concatenated after base instruction.
  • Project = name + local path. All paths exposed to the CLI; no single cwd.
  • Model = user-registered command template (shell-out, not API). See ADR-0001.
  • Template IO: input via stdin, result from stdout, stderr → logs.
  • Placeholders: {system}, {input}, {projects}, {projects:--flag}.
  • Task = title + body; targets a flow OR a single agent; holds status/results.
  • Failure: retry N times (Settings) then stop + mark failed; keep partial.
  • Concurrency: multiple tasks run at once; steps within a flow are sequential.
  • Persistence: per-step (input/output/exit/timing) + final result.
  • Delete: blocked while referenced (shows what uses it).
  • Stack: Next.js App Router · Bun (pm + runtime) · bun:sqlite · SSE streaming.
  • MVP pages: Skills · Projects · Models · Agents · Flows · Tasks · Settings.

Open Questions

None blocking. Two tunables were defaulted rather than left open — both live in Settings and are trivially changeable:

  • Retries default = 1 (retry a failed step once, then stop + fail).
  • Per-step timeout default = 600s (kill a hung CLI; a step exceeding it counts as a failure and is retried per the retry rule).

Flag now if either default is wrong — otherwise implementation proceeds with these.

Global Constraints

  • Runtime: Bun for install and run (bun install, bun --bun run dev). Language: TypeScript, strict.
  • DB: bun:sqlite only. DB file at ~/.orkestra/orkestra.db (dir auto-created). Tests use an in-memory / temp-file DB.
  • No shell interpretation when spawning models — build an argv array and use Bun.spawn(argv, …). Never pass user text through sh -c.
  • Tests: bun test. The engine is tested against a fake CLI fixture (a shell script that echoes), never a real LLM.
  • Every task is TDD: failing test → run (see it fail) → minimal impl → run (pass) → commit. Commit messages plain, no AI co-author.
  • Each entity: names are unique (case-insensitive) within their type.

How a run works

Data model (SQLite)

-- definitions
skills(id, name UNIQUE, body, created_at, updated_at)
projects(id, name UNIQUE, path, created_at, updated_at)
models(id, name UNIQUE, command, created_at, updated_at)      -- command = template string
agents(id, name UNIQUE, base_instruction, model_id → models, created_at, updated_at)
agent_skills(agent_id → agents, skill_id → skills, position)  -- ordered; PK(agent_id, skill_id)
agent_projects(agent_id → agents, project_id → projects)      -- PK(agent_id, project_id)
flows(id, name UNIQUE, created_at, updated_at)
flow_steps(id, flow_id → flows, agent_id → agents, position)  -- ordered

-- work + history
tasks(id, title, body, target_type ['flow'|'agent'], target_id, status
      ['pending'|'running'|'succeeded'|'failed'], created_at, updated_at)
runs(id, task_id → tasks, status, final_output, error, started_at, finished_at)
run_steps(id, run_id → runs, position, agent_id, agent_name, input, output,
          exit_code, error, status, started_at, finished_at)

-- config (singleton row id=1)
settings(id=1, retries DEFAULT 1, step_timeout_seconds DEFAULT 600)

Deletes on skills/projects/models/agents/flows are blocked when referenced by the relation/step/agent/task tables (checked in repo layer, surfaced to the UI).

// lib/db/index.ts
import { Database } from "bun:sqlite";
import { homedir } from "os";
import { mkdirSync } from "fs";
import { join } from "path";
import { MIGRATIONS } from "./migrations";
export function openDb(path?: string) {
  let file = path;
  if (!file) { const dir = join(homedir(), ".orkestra");
    mkdirSync(dir, { recursive: true }); file = join(dir, "orkestra.db"); }
  const db = new Database(file);
  db.exec("PRAGMA journal_mode = WAL; PRAGMA foreign_keys = ON;");
  for (const stmt of MIGRATIONS) db.exec(stmt);
  return db;
}
let _db: import("bun:sqlite").Database | null = null;
export function db() { return (_db ??= openDb()); }

Test strategy summary

Unit

Template engine (T8), repos (T3–T7). Pure functions + :memory: DB. Fast, exhaustive on edge cases.

Integration

Executor + runner (T9–T10) against the echo fixture — real process spawning, zero LLM, deterministic.

E2E

Actions + full run (T11, T18). Browser verification of the Tasks streaming view. Reviewer confirms.
